# Own the desktop config in dotfiles-rheniite (replace zdots)

- **Status:** proposed
- **Created:** 2026-07-14
- **Area:** image (`Containerfile`) + `dotfiles-rheniite` (separate repo)
- **Related:** prerequisite/partner of
  [stabilize-git-packages.md](stabilize-git-packages.md) — pinning DMS/niri to
  stable is only coherent if the *config* is pinned/owned too. zdots =
  `github.com/zirconium-dev/zdots` (submodule at
  `mkosi.extra/usr/share/zirconium/zdots`, baked to `/usr/share/zirconium/zdots`).

## Problem

Zirconium's default desktop config (**zdots**) is authored against **git-HEAD DMS/niri**,
is **owned by the image**, and is **re-applied to `$HOME` daily**:

- `chezmoi-init.service` (on `graphical-session-pre`) runs
  `chezmoi apply -S /usr/share/zirconium/zdots`.
- `chezmoi-update.timer` fires `OnBootSec=5m` then `OnUnitInactiveSec=1d`, re-running
  `chezmoi apply … -S /usr/share/zirconium/zdots` with `yes s` (skips files you've
  locally modified, **re-applies everything else** from the baked copy).

We currently consume this: the image owns `~/.config/niri/config.kdl`, and
dotfiles-rheniite ships only `local.kdl` + `local/*.kdl` overrides. The moment
DMS/niri are pinned to **stable** (see the partner item) while zdots keeps tracking
git, the shipped config skews against the installed binaries — DMS `settings.json`
for a newer schema, `config.kdl` using post-`26.04` niri options (niri can refuse to
load it), `dms ipc` keybinds calling renamed verbs — and the daily timer re-imposes
the mismatch. Going stable coherently means **owning the config layer**, not just
swapping RPMs.

## What's actually inside zdots (researched 2026-07-14)

The niri config is a 3-layer include chain, only part of which lives in `$HOME`:

```
~/.config/niri/config.kdl                 (zdots, "DO NOT EDIT")
├─ include "dms.kdl"                       → dms/*.kdl  (mostly GENERATED BY DMS at runtime)
├─ include "/usr/share/zirconium/zdots/system/niri/zirconium.kdl"   (BAKED IN IMAGE, not $HOME)
│    └─ includes/{binds,input,layout,window-rules,misc,shadow,dms-base}.kdl
└─ include optional=true "local.kdl"       (dotfiles-rheniite already owns this)
```

Ownership verdict per file:

| zdots content | ~Lines | Category | Action |
|---|---|---|---|
| `system/niri/includes/*` (binds, input, layout, window-rules, misc, **dms-base**, shadow) | ~265 | niri base config | **Own** — but `local/*.kdl` (341 lines) already parallels most; net-new ≈ `dms-base`(16: quickshell backdrop + lid-lock), `misc`(4), `shadow`(13) + small deltas |
| `niri/config.kdl` + `dms.kdl` | 26 | entry + DMS include glue | **Own** — write our own entry, drop the `/usr/share/zirconium/zdots` include |
| `niri/dms/binds.kdl` + `dms/colors.kdl` | 73 | **DMS-IPC keybinds** + color seed | **Own** — coupled to DMS ipc verbs (spotlight/clipboard/settings/lock/notepad/…) |
| `DankMaterialShell/settings.json` + `clsettings.json` | 226 | **DMS settings schema** | **Own a copy** matched to stable DMS 1.5.0 (copy the 1.5.0 default, tweak) |
| all matugen outputs (gtk css ×2, `qt6ct/matugen.conf`, `DankMatugen*.colors`, `dgop/colors.json`, `dms-colors.json`) | ~700 | **matugen-generated theming** | **Don't hand-maintain** — we already own `matugen/` config+templates; let it regenerate |
| `niri/dms/{alttab,layout,wpblur,outputs,cursor,windowrules}.kdl` | — | **DMS-generated at runtime** (GUI writes them) | **Don't vendor** |
| `state/…/session.json`, `cache/…/cache.json` | ~114 | DMS runtime state | **Don't vendor** |
| `foot/*`, `symlink_vesktop`, `fastfetch` | ~45 | apps we don't use (kitty) | **Drop** |

**Net-new authoring ≈ one focused session:** our own `config.kdl` (~15 lines) + the
zdots-only niri deltas (~40, `dms-base.kdl` being the one that matters) + the
niri↔DMS glue (~70) + a copied `settings.json` (~226). The old
`dotfiles-bluefin-niri` already vendored `config.kdl` + `dms/*.kdl`, so this is
re-treading known ground.

## Options

### A. Own it in dotfiles-rheniite + neutralize the image's zdots (recommended)

Mask the base's chezmoi units in the `Containerfile`; move the "Own" rows into
dotfiles-rheniite; ship our own `config.kdl` that includes our topic files directly.

- ✅ Full control; config matched to pinned stable DMS/niri; no daily re-imposition;
  no `/usr/share/zirconium/zdots` dependency.
- ⚠️ We port upstream config improvements manually (acceptable — the point of pinning
  is a *deliberate* cadence).

### B. Replace the baked zdots content in the image

Keep Zirconium's chezmoi mechanism, but `COPY` our own dots tree over
`/usr/share/zirconium/zdots` in the `Containerfile`.

- ✅ Reuses the existing deploy/update machinery.
- ❌ We maintain a zdots-shaped tree inside the *image* repo (awkward — config belongs
  in dotfiles-rheniite), and the daily timer still fights any runtime edits.

### C. Pin the zdots submodule to a matching commit (freeze, don't own)

In the Zirconium fork, pin the `zdots` submodule to a commit contemporaneous with DMS
1.5.0. Config + binaries become a coherent frozen set with near-zero authoring.

- ✅ Least effort; keeps upstream's config as-is.
- ❌ No divergence/personalisation beyond `local.kdl`; you bump zdots + packages
  together and inherit upstream's choices. Good stopgap, not full ownership.

## Recommendation

**Option A**, done together with [stabilize-git-packages.md](stabilize-git-packages.md)
so binaries *and* config move to stable as one matched set. Use **C** only as an
interim if A's authoring isn't worth it yet. The one artifact that stays coupled
either way is **`lader-dank`** (our own DMS *plugin* under
`~/.config/DankMaterialShell/plugins`) — it tracks whatever DMS API we pin, so it's a
one-time port to 1.5.0, then stable.

## Implementation sketch

### 1. Stop the base from deploying/re-applying zdots (`Containerfile`)

```dockerfile
# dotfiles-rheniite owns the desktop config; disable Zirconium's zdots deploy/update
# so it can't re-impose git-DMS config over our pinned-stable setup on a daily timer.
RUN systemctl --global mask \
      chezmoi-init.service chezmoi-update.service chezmoi-update.timer
```

(Alternatively drop the `01-zirconium.preset` user-preset entries that enable them,
and/or remove `/usr/share/zirconium/zdots` — but masking is the smallest, clearest.)

### 2. Move the "Own" rows into dotfiles-rheniite

- `dot_config/niri/config.kdl` — **our** entry (see skeleton below), replacing the
  image's. No `/usr/share/zirconium/zdots` include.
- Fold zdots' `system/niri/includes/{dms-base,misc,shadow}.kdl` (+ any input/layout/
  window-rules deltas) into `dot_config/niri/local/` topic files.
- `dot_config/niri/dms.kdl` + `dot_config/niri/dms/binds.kdl` + `dms/colors.kdl`
  (the DMS-IPC glue), matched to DMS 1.5.0's ipc verbs.
- `dot_config/DankMaterialShell/settings.json` + `clsettings.json` — copy the stable
  DMS 1.5.0 defaults, then tweak.
- **Drop** `foot/*`, `symlink_vesktop`; let `matugen` regenerate all color files
  (we already ship `matugen/` config+templates); do **not** vendor `session.json`,
  `cache.json`, or the DMS-generated `dms/*.kdl`.

### 3. New `~/.config/niri/config.kdl` skeleton (ours)

```kdl
// dotfiles-rheniite owns this now (was zdots). No /usr/share/zirconium include.
include "dms.kdl"                       // DMS integration glue (ours)
include "local.kdl"                     // our topic files (settings/startup/input/layout/binds/window-rules)
// dms-base / misc / shadow now live inside local/*.kdl
```

### 4. Port `lader-dank` to the pinned DMS

Rebuild/verify the plugin against DMS 1.5.0's plugin API; update
`run_once_after_install-dank-lader-plugin.sh` if the plugin layout changed.

## Verification

- Fresh `chezmoi apply` on a Rheniite test host: `niri validate` accepts `config.kdl`
  (no unknown-option errors); DMS launches with our `settings.json` (dankbar, theme).
- Every `dms ipc` keybind works (spotlight, clipboard, settings, lock, notepad,
  notifications, dankdash) — these are the DMS-version-coupled binds.
- `matugen` regenerates the color files (change wallpaper/theme → gtk/qt/kde recolor).
- `dank-lader` (Mod+F12) plugin loads and drives DMS.
- Confirm the base no longer touches `$HOME`: `systemctl --user status
  chezmoi-update.timer` is `masked`; nothing reverts our niri/DMS config after 24h.
- `bootc container lint` passes; image builds.
